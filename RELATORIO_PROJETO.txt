RELATÓRIO BASE DO PROJETO
Raquel Maia Massoterapeuta — SPA estática de agendamento

1. VISÃO GERAL

Este projeto é uma SPA estática para agendamento de sessões de massoterapia, desenvolvida com:
- HTML
- CSS
- JavaScript vanilla com ES Modules
- Hash routing
- Firebase via CDN/ESM no navegador

O sistema foi pensado para rodar em hospedagem simples, como GitHub Pages ou qualquer servidor estático, sem dependências de build, sem npm e sem framework.

O foco atual do projeto é entregar um MVP funcional e utilizável, com:
- experiência pública de agendamento;
- painel administrativo autenticado;
- bloqueio real de agenda;
- persistência em Firebase Realtime Database;
- autenticação do admin com Firebase Auth Email/Senha.

2. OBJETIVO DO SISTEMA

O sistema permite que clientes:
- escolham um serviço;
- escolham uma data;
- escolham um horário disponível;
- preencham nome e telefone;
- enviem uma solicitação de agendamento;
- confirmem o envio e recebam opção de contato via WhatsApp.

Ao mesmo tempo, permite que a profissional:
- entre no painel administrativo;
- visualize agendamentos por status;
- confirme, rejeite, cancele ou exclua;
- configure os dias e horários de atendimento;
- defina exceções por dia da semana;
- acione rapidamente o WhatsApp do cliente com mensagens prontas.

7. FLUXO PÚBLICO DE AGENDAMENTO

Etapa 1 — Escolha do serviço
Na tela da profissional, o usuário:
- visualiza os serviços;
- pode pesquisar pelo nome;
- seleciona um serviço agendável;
- é direcionado à escolha de data e horário.

Alguns itens são institucionais e não necessariamente agendáveis.

Etapa 2 — Escolha de data
Na tela #/agendar, o usuário:
- vê os próximos dias válidos;
- escolhe um dia disponível;
- carrega os horários possíveis para aquele serviço.

A lista de datas respeita:
- dias permitidos na configuração;
- agenda padrão;
- exceções por dia da semana;
- regras de almoço e expediente.

Etapa 3 — Escolha de horário
O usuário seleciona apenas o horário inicial.

A aplicação calcula internamente a duração completa da sessão em blocos de 15 minutos.

Exemplo:
- serviço de 60 minutos;
- início às 14:30;
- bloqueios internos: 14:30, 14:45, 15:00, 15:15.

Logo:
- horários que colidem com esse intervalo deixam de ser válidos;
- o próximo início real disponível passa a ser o primeiro que não sobrepõe a sessão anterior.

Etapa 4 — Confirmação dos dados
Na tela #/confirmar, o usuário informa:
- nome;
- telefone.

E visualiza:
- serviço;
- data;
- horário;
- duração;
- valor.

Ao confirmar:
1. o sistema tenta reservar o bloco completo no Firebase;
2. se houver conflito, a interface informa de forma amigável;
3. se der certo, o booking é criado com status pending.

Etapa 5 — Sucesso
Na tela #/sucesso, o usuário vê:
- resumo do agendamento;
- opção de contato por WhatsApp.

A mensagem foi ajustada para indicar que o cliente reservou um horário pelo site, e não que o agendamento já foi confirmado.

8. LÓGICA DE AGENDA E BLOQUEIO DE HORÁRIOS

Granularidade
A agenda trabalha com fatias internas de 15 minutos.

Regra principal
O cliente escolhe apenas o início.
O sistema bloqueia automaticamente todo o intervalo da sessão.

Cálculo de duração
A lógica considera:
- intervalMinutes = 15
- requiredSlots = Math.ceil(duracaoServico / 15)

Exemplo:
- 60 min = 4 slots
- 90 min = 6 slots

Disponibilidade
Um horário inicial só aparece se todos os slots contíguos necessários estiverem livres.

O sistema não mostra inícios que:
- cruzem o almoço;
- ultrapassem o fim do expediente;
- caiam em horários bloqueados;
- colidam com reservas pending ou confirmed;
- colidam com reservas já existentes no intervalo da sessão.

Resultado visual
A grade de horários tende a mostrar apenas:
- horários realmente agendáveis;
- horários coerentes com a duração do serviço;
- sem confusão entre “ocupado” e “inválido por regra”.

9. PERSISTÊNCIA E FIREBASE RTDB

Estrutura geral
Bookings
Os agendamentos ficam em:
- /bookings/{bookingId}

Locks de agenda
Os bloqueios ficam em:
- /slotLocks/raquel-maia/{YYYY-MM-DD}

Dentro do dia, cada fatia de 15 minutos fica registrada.

Estratégia de segurança
O sistema não trava apenas um horário.
Ele faz uma transação no nó do dia inteiro, garantindo que:
- a faixa completa seja validada de forma atômica;
- não haja escrita parcial;
- não haja dupla reserva concorrente;
- duas pessoas não fechem o mesmo intervalo ao mesmo tempo.

Fluxo de confirmação
Ao criar um booking:
1. o sistema calcula todos os slots necessários;
2. executa transação no dia correspondente;
3. se todos estiverem livres, grava os bloqueios;
4. em seguida cria o agendamento com status pending;
5. se a escrita do booking falhar, faz rollback dos slots.

Status que bloqueiam a agenda
Bloqueiam disponibilidade:
- pending
- confirmed

Liberam disponibilidade:
- rejected
- cancelled
- exclusão do agendamento

10. PAINEL ADMINISTRATIVO

Acesso
A rota #/admin é protegida com:
- Firebase Auth
- login por e-mail e senha

Comportamento
- usuário não autenticado: vê formulário de login
- usuário autenticado: vê dashboard administrativo

Abas principais
Agenda
Exibe agendamentos agrupados por status:
- Pendentes
- Confirmados
- Rejeitados
- Cancelados

Config
Permite configurar:
- dias de atendimento;
- horário padrão de abertura;
- horário padrão de encerramento;
- intervalo de almoço;
- intervalo entre horários;
- modo sem intervalo;
- exceções por dia da semana.

Ações administrativas
Dependendo do status, o admin pode:
- Confirmar
- Rejeitar
- Cancelar
- Excluir

Regras
- Confirmar: mantém o lock
- Rejeitar: libera o lock
- Cancelar: libera o lock
- Excluir: remove o booking e também limpa os locks relacionados

13. REGRAS DO FIREBASE

O arquivo database.rules.json.example já prevê:
- leitura pública de slotLocks;
- leitura de bookings apenas para admin autenticado;
- indexação por criadoEm;
- validação mínima de estrutura dos bookings;
- leitura pública de config;
- escrita de config apenas com autenticação.

Ponto importante
As regras reais precisam ser publicadas no Firebase Console.
Se o console estiver com regras abertas ou diferentes, o comportamento pode divergir.
